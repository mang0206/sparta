- 클라우드 환경(AWS/GCP/Azure)에서 컨테이너가 어떻게 활용되는지 조사
1. AWS (Amazon Web Services)
- EKS (Elastic Kubernetes Service): 표준 쿠버네티스 환경을 AWS에서 관리형으로 제공
- ECS (Elastic Container Service): AWS 전용 컨테이너 오케스트레이션 도구 쿠버네티스보다 설정이 단순하고 AWS의 다른 서비스(ELB, IAM 등)와 통합이 매우 강력
- Fargate (서버리스): EC2 인스턴스(서버)를 직접 관리할 필요 없이 컨테이너만 띄우면 되는 서버리스 엔진. EKS와 ECS 모두에서 사용 가능
- ECR (Elastic Container Registry): 도커 컨테이너 이미지를 저장하고 관리하는 저장소

2. GCP (Google Cloud Platform)
- GKE (Google Kubernetes Engine): 클라우드 시장에서 가장 고도화되고 안정적인 관리형 쿠버네티스 서비스. 자동화된 업데이트와 복구 기능이 탁월
- Cloud Run (서버리스): 완전 관리형 서버리스 컨테이너 서비스. HTTP 요청을 처리하는 웹 서비스 배포에 최적화
- Artifact Registry: 컨테이너 이미지 및 언어 패키지를 통합 관리하는 저장소.

3. Azure (Microsoft Azure)
엔터프라이즈 환경 및 Windows 기반 컨테이너 지원이 강력
- AKS (Azure Kubernetes Service): 마이크로소프트의 관리형 쿠버네티스 서비스
- Active Directory(AD) 등 MS 엔터프라이즈 솔루션과의 연동이 강력
- Azure Container Apps (서버리스): 쿠버네티스 기반이지만 복잡한 설정 없이 마이크로서비스를 구축할 수 있게 해주는 서버리스 컨테이너 서비스
- ACI (Azure Container Instances): 오케스트레이션 없이 단일 컨테이너를 가장 빠르게 실행할 수 있는 서비스
- ACR (Azure Container Registry): 도커 레지스트리 관리 서비스


- VM과 Docker의 차이점 정리
| 비교 항목 | VM (Virtual Machine)  | Docker (Container) |
|------------------|--------------------------------------------------------|-----------------------------------------------------|
| 격리 대상        | 하드웨어 (Hardware)                                    | 운영체제 (OS / Kernel)                              |
| 작동 원리        | Hypervisor 위에 각각 Guest OS 설치                    | Host OS의 Kernel을 공유하며 컨테이너 엔진 구동      |
| 무게(용량)       | 무거움                            | 가벼움                          |
| 부팅 속도        | 느림 (OS 부팅 필요, 수 분 소요)                       | 빠름 (프로세스 실행 수준, ms ~ 초 단위)             |
| 이식성           | 낮음 (OS/환경 설정 차이에 영향 받음)                  | 높음 (이미지 그대로 가져가면 어디서든 거의 동일)   |
| 보안(격리성)     | 강함 (각 VM이 완전 분리된 독립 OS)                    | 상대적으로 약함 (커널 공유로 커널 취약점 영향 가능) |


- 아래 질문에 대한 답변을 정리하여 제출
1. Docker의 주요 개념은 무엇인가?
- 이미지 (Image):
애플리케이션 실행에 필요한 코드, 런타임, 라이브러리, 설정 등을 모두 포함한 실행 가능한 애플리케이션 패키지

- 컨테이너 (Container):
이미지를 바탕으로 실행된 '실제 프로세스'

- 도커파일 (Dockerfile):
도커 이미지를 만들기 위한 '설정 스크립트 파일'


- 레지스트리 (Registry):
도커 이미지를 저장하고 배포하는 저장소

2. VM과 컨테이너의 차이점은 무엇인가?
VM (Virtual Machine)
구조: 하이퍼바이저 위에서 각 VM이 자체 게스트 OS를 가짐
특징: 하드웨어 전체를 가상화하여 완벽한 격리, 다른 OS 사용 가능, 이미지 크기 큼 (수 GB), 시작 속도 느림 (수십 초)
장점: 뛰어난 격리성, 다양한 OS 환경 지원
단점: 무겁고 느림, 리소스 소모 많음
적합한 경우: 서로 다른 OS가 필요한 서비스, 높은 수준의 격리가 중요한 경우

컨테이너 (Container)
구조: 호스트 OS 커널을 공유하며 앱과 라이브러리만 패키징
특징: 앱 실행 환경만 가상화, 이미지 크기 작음 (MB 단위), 시작 속도 빠름 (즉시)
장점: 가볍고 빠름, 리소스 효율성 높음, 배포 및 확장 용이 (일관된 환경)
단점: 호스트 OS와 커널 공유로 VM보다 격리성 낮음 (보안 취약점 가능성), 동일 OS 환경에서만 작동
적합한 경우: 마이크로 서비스, 빠른 배포 및 확장, 개발/운영 환경 일관성 확보.

핵심 차이
격리 vs 효율: VM은 격리, 컨테이너는 효율
OS: VM은 독립 OS, 컨테이너는 호스트 OS 공유
속도/크기: VM은 느리고 큼, 컨테이너는 빠르고 작음


3. 클라우드에서 Docker를 사용하는 이유는 무엇인가?
- 이식성 (Portability):
개발자의 로컬, 테스트 서버, 클라우드 운영 서버 어디서든 동일한 환경과 동작 보장

- 자원 효율성 및 비용 절감:
VM보다 훨씬 적은 자원(CPU/RAM)으로 더 많은 애플리케이션을 구동 가능. 이는 클라우드 인프라 비용 절감으로 직결

- 빠른 확장성 (Auto Scaling):
트래픽이 급증할 때 VM은 부팅에 몇 분이 걸리지만, 컨테이너는 밀리초(ms) 단위로 생성되어 즉각적인 대응(Scale-out)이 가능

- MSA (Microservices Architecture) 최적화:
거대한 앱을 작은 기능 단위로 쪼개어 배포하고 관리하기에 최적화되어잇어 각 기능을 독립적으로 업데이트하거나 배포 가능
